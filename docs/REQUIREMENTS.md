## 1. 项目概述

**話窓 Hanamado** 是一款纯网页架构的日语学习工具，通过 AI 角色对话提升日语能力。

### 核心特色

- **AI 回复自动分词**：每个词为独立可交互单元，支持点击查词与选中句子询问语法。
- **多角色情景对话**：可切换不同角色与难度进行情景对话，模拟真实语言环境。
- **苹果风格设计**：界面采用 Apple 设计语言，简洁优雅。
- **三栏布局**：左侧功能/聊天列表，中间对话区，右侧按需弹出的知识面板（类似 Trae Solo 布局）。

### 目标用户

- 日语学习者（初级至高级）
- 希望通过情景对话提升口语和阅读理解能力的用户
- 偏好轻量级、无需安装的网页工具的用户

## 2. 技术约束

| 类别 | 约束 |
|------|------|
| **架构** | 纯前端网页，不使用自建后端服务器 |
| **部署平台** | [Vercel](https://vercel.com)（支持静态站点及 Serverless Functions 代理） |
| **数据持久化** | 浏览器本地存储（localStorage / IndexedDB） |
| **AI 调用** | 前端直接请求用户配置的 API（OpenAI 兼容接口），或通过 Vercel Functions 代理以隐藏密钥 |
| **推荐方案** | Vercel Edge Functions 代理方式（安全性更高），同时保留纯前端直连选项 |

### 技术选型建议

- **框架**：React（推荐）或 Vue 3，采用组件化架构
- **样式**：Tailwind CSS + 自定义 CSS Module / 毛玻璃效果
- **状态管理**：React Context + useReducer 或 Zustand
- **Markdown 渲染**：react-markdown 或 marked
- **离线词典**：JMdict JSON 格式

## 3. UI/UX 设计总则

### 3.1 设计语言

- **风格**：Apple 风格 — 简洁、大量留白、圆角卡片、柔和阴影、毛玻璃效果（glassmorphism）
- **字体**：优先使用系统原生字体
  - 中文/日文：PingFang SC / Hiragino Sans
  - 英文/数字：San Francisco / Inter
- **图标**：SF Symbols 风格线性图标（如 Lucide Icons）

### 3.2 布局结构（三栏式）

```
┌──────────────┬──────────────────────┬─────────────────┐
│              │                      │                 │
│   左侧栏     │     中央聊天区       │   右侧面板      │
│  (Sidebar)   │    (Chat Area)       │  (Knowledge     │
│              │                      │   Panel)        │
│  · 功能图标  │  · 消息气泡          │                 │
│  · 会话列表  │  · 分词语块          │  · 字典模式     │
│              │  · 输入区域          │  · 语法模式     │
│              │                      │                 │
└──────────────┴──────────────────────┴─────────────────┘
```

- **左侧栏**：固定宽度 ~260px，包含顶部功能图标和下方聊天会话列表
- **中央聊天区**：弹性宽度，消息气泡展示，AI 回复内容以分词块渲染
- **右侧面板**：默认隐藏，由用户交互触发后从右侧滑入，宽度 ~360px，采用覆盖滑出或挤压布局

### 3.3 配色方案

- **浅色主题**（默认）：
  - 背景：`#f5f5f7`（Apple 浅灰）
  - 卡片：`rgba(255,255,255,0.8)` + 毛玻璃
  - 主文字：`#1d1d1f`
  - 强调色：`#007aff`（Apple Blue）
- **深色主题**（可选切换）：
  - 背景：`#1c1c1e`
  - 卡片：`rgba(44,44,46,0.8)`
  - 主文字：`#f5f5f7`

### 3.4 响应式适配

- **桌面端**（≥1024px）：完整三栏布局
- **平板端**（768px–1023px）：左侧栏可折叠，右侧面板覆盖显示
- **移动端**（<768px）：单栏布局，左侧栏和右侧面板均为全屏覆盖层
- 基础可用性优先保证，桌面端为主要设计目标

## 4. 功能需求详述

### 4.1 左侧栏 — 会话管理

#### 4.1.1 功能菜单

顶部区域使用图标按钮展示：

| 图标 | 功能 | 说明 |
|------|------|------|
| ➕ | 新建聊天 | 打开新建聊天对话框 |
| ⚙️ | API 配置 | 进入 API 配置页面 |
| ℹ️ | 关于 | 显示应用信息、版本号 |

#### 4.1.2 聊天会话列表

每项会话卡片展示：

- **角色名称**（如：便利店店员）
- **难度标签**（初级 / 中级 / 高级），使用不同颜色区分
- **最后一条消息预览**（截断至一行，灰色文字）
- **时间戳**（相对时间，如"3分钟前"）

**交互行为**：

- **点击**：切换到对应聊天会话
- **右键 / 长按**：弹出删除确认菜单
- 当前活跃会话高亮显示

#### 4.1.3 数据持久化

- 所有聊天记录保存在 localStorage（小数据）或 IndexedDB（大数据）
- 刷新页面不丢失会话数据
- 定期自动保存，避免数据丢失
- 注意 JSON 序列化时的转义处理和存储容量限制（localStorage 通常 5–10MB）

### 4.2 中央聊天区

#### 4.2.1 消息渲染

**用户消息**：

- 普通文本气泡，右对齐
- 气泡背景：`#007aff`（蓝色），文字白色
- 圆角：左上、左下、右下为 16px，右上为 4px

**AI 消息**：

- 左对齐布局
- **分词句子**：每个词为独立可交互单元，以圆角标签/底纹块展示，间距 4–6px
- **中文翻译**：位于分词句子下方，较小字号（13px），灰色（`#86868b`），整体展示
- 分词句子需根据 AI 返回的 `tokens` 数组渲染

**消息状态**：

- 发送中：显示加载动画（三个点跳动）
- 发送失败：显示错误图标 + "重试"按钮
- AI 响应格式异常：显示"AI 响应格式异常，请重试"提示

#### 4.2.2 分词交互

**Hover（悬停）**：

- 词块上出现底纹高亮（轻微背景色变化）
- 显示浮层 Tooltip（可展示读音、基本义项，如可从本地词典快速匹配）

**Click（点击单词）**：

- 触发右侧面板弹出
- 自动切换到"字典"模式
- 展示该词的详细解释（读音、词性、释义、例句等）

#### 4.2.3 句子语法查询

**触发流程**：

1. 用户在 AI 回复或自己的消息中**滑动选择一段文字**
2. 选中文字后，在选中区域附近出现**浮动菜单按钮**："询问语法"
3. 点击后，**右侧面板弹出**并切换到"语法解析"模式
4. 将选中的句子作为上下文发送给 AI
5. AI 返回语法解释，以 Markdown 格式渲染展示

**注意**：移动端需适配触摸选择交互。

#### 4.2.4 输入区域

- 底部固定输入框（sticky 定位）
- 支持 **Enter 发送**，**Shift+Enter 换行**
- 输入框高度自适应（最小 1 行，最大 5 行）
- 发送后自动清除输入内容
- 右侧可附发送按钮（纸飞机图标）

#### 4.2.5 空状态

- 无聊天记录时显示欢迎引导
- 提示用户创建新会话或选择已有会话
- 可展示简短的使用引导

### 4.3 右侧知识面板

#### 4.3.1 显示模式

两种模式由触发动作自动切换：

| 模式 | 触发方式 | 内容 |
|------|----------|------|
| **字典模式** | 点击 AI 消息中的单词 | 单词详细信息（读音、词性、释义、例句） |
| **语法模式** | 选中句子 → "询问语法" | AI 语法解释（Markdown 渲染） |

#### 4.3.2 面板交互

- **关闭方式**：
  - 点击面板顶部 × 关闭按钮
  - 点击面板外的遮罩区域
  - 再次触发新的查词/语法查询（替换当前内容）
- **宽度**：~360px，可适当拖拽调整
- **布局方式**：覆盖滑出（overlay + slide-in），或挤压布局（squeeze），不遮挡主要聊天区
- **动画**：从右侧滑入/滑出，带缓动效果（ease-out，300ms）

#### 4.3.3 字典模式内容

- 单词原形（辞書形）
- 读音（平假名 / 罗马音）
- 词性（名詞、動詞、形容詞 等）
- 中文释义
- 常见用法 / 例句
- 若本地词典未命中，显示"词典未收录，可点击通过 AI 查询"提示

#### 4.3.4 语法模式内容

- 以 Markdown 格式渲染 AI 返回的语法解释
- 包含：语法点分析、用法说明、类似表达、例句
- 加载状态：显示骨架屏或加载动画
- 错误状态：显示错误信息 + 重试按钮

### 4.4 新建聊天对话框

#### 4.4.1 触发方式

- 点击左侧栏 ➕ 按钮
- 空状态下的引导按钮

#### 4.4.2 配置项

**角色选择**（预设 4–6 个角色，初期）：

| 角色 | 用语习惯 | 场景 |
|------|----------|------|
| 便利店店员 | 敬语（です・ます体） | 日常购物 |
| 大学朋友 | 简语（だ・である体） | 校园闲聊 |
| 面试官 | 敬语（正式） | 求职面试 |
| 居酒屋老板 | 简语 + 行业用语 | 餐饮社交 |
| 旅行向导 | 敬语 | 旅游问路 |
| 同事 | 敬语（适度） | 职场交流 |

每个角色拥有不同的用语习惯（敬语/简语、行业词汇等）。

**难度选择**：

| 难度 | AI 用词 | 语法复杂度 | 回复长度 |
|------|---------|-----------|----------|
| 初级 | N5–N4 基础词汇 | 简单句型 | 1–2 句 |
| 中级 | N3–N2 词汇 | 中等复杂句 | 2–3 句 |
| 高级 | N1+ 词汇 | 复杂句/敬语多变 | 3–5 句 |

**谁先开口**：

- **用户先开口**：聊天区显示空态，等待用户输入第一条消息
- **AI 先开口**：创建成功后 AI 自动发送第一条引导消息（根据角色+难度生成开场白），用户接话

#### 4.4.3 确认后行为

- 生成新会话 ID
- 保存到本地存储
- 切换到该聊天会话
- 若选择"AI 先开口"，立即调用 AI 生成开场白

### 4.5 用户 API 配置页面

#### 4.5.1 入口

- 左侧栏 ⚙️ 图标
- 独立页面或全屏弹窗

#### 4.5.2 配置项

| 字段 | 类型 | 说明 |
|------|------|------|
| API 基础 URL | 文本输入 | 支持 OpenAI 格式及兼容接口（如 `https://api.openai.com/v1`） |
| API Key | 密码框 | 密钥仅保存在本地浏览器，需注明安全性 |
| 模型名称 | 文本输入 | 如 `gpt-4o-mini`、`deepseek-chat`、`gpt-4o` 等 |
| Temperature | 滑块/数字 | 0–2，默认 0.7 |
| Max Tokens | 数字输入 | 默认 1024 |

#### 4.5.3 功能

- **"测试连接"按钮**：发送简单请求验证配置是否有效，显示成功/失败结果
- **保存配置**：加密存储于 localStorage（至少 Base64 编码，避免明文泄露），下次使用自动填充
- **安全提示**：醒目警告 — 直接从前端调用 API 会暴露密钥，建议开发者通过 Vercel Edge Functions 代理请求

#### 4.5.4 代理模式（可选）

- 提供 Vercel Functions 代理选项
- 配置 `/api/chat` 路由，服务端转发请求并注入 API Key
- 前端不直接持有密钥，仅通过代理调用
- 需在 `vercel.json` 中配置环境变量

## 5. AI 对话与数据格式规范

### 5.1 请求与上下文构建

#### 5.1.1 System Prompt 结构

系统提示词需包含以下要素：

1. **角色设定**：根据所选角色，设定身份、语气、用语习惯
2. **难度要求**：控制用词复杂度、语法难度和回复长度
3. **输出格式要求**：严格说明必须返回 JSON 格式

**System Prompt 模板示例**：

```
你是一个日语对话伙伴，请按以下设定进行对话：

【角色】：{角色名}
【用语习惯】：{敬语/简语等描述}
【场景】：{场景描述}
【难度】：{难度等级}
- 用词范围：{对应JLPT等级}
- 回复长度：{句数范围}
- 语法复杂度：{描述}

【重要】你必须严格按照以下 JSON 格式回复，不要包含任何其他内容：
{
  "message": "日语完整句子",
  "tokens": ["词1", "词2", "词3", "标点"],
  "translation": "中文翻译"
}

注意：
- message 是完整的日语句子，不含分词标记
- tokens 按自然分词粒度拆分，标点符号单独作为元素
- translation 是整句对应的中文翻译
- 每次只回复一句日语
```

#### 5.1.2 上下文管理

- 携带最近 N 轮对话历史（建议 10–20 轮）
- 超出限制时截断较早的历史
- 保留 System Prompt 完整性
- 估算 Token 使用量，避免超出模型限制

### 5.2 AI 回复 JSON 格式规范

每次 AI 对话响应（包括 AI 先开口的初始消息）都**必须**是一个合法的 JSON 对象：

```json
{
  "message": "こんにちは！お元気ですか？",
  "tokens": ["こんにちは", "！", "お元気", "です", "か", "？"],
  "translation": "你好！你还好吗？"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `message` | string | 完整的日语句子（纯文本，不含分词标记） |
| `tokens` | string[] | 按自然分词粒度拆分的数组（需包含标点等） |
| `translation` | string | 整句对应的中文翻译 |

**设计原则**：

- **优先采用单句模式**：每次 AI 只回复一句日语，保证分句交互清晰
- 若必须回复多句，可约定返回数组格式，但需提前评估解析复杂度
- 前端需进行 JSON 解析校验，格式错误时：
  - 向用户提示"AI 响应格式异常"
  - 允许重试
  - 记录错误日志

### 5.3 语法查询请求

用户选中句子点击"询问语法"时，单独发送一条消息：

- **System Prompt**：调整为日语教师角色
- **要求**：用中文解释该句子的语法点、用法、类似表达等
- **返回格式**：普通文本（支持 Markdown），**不强制 JSON**

**语法查询 System Prompt 模板**：

```
你是一位专业的日语教师。请用中文解释以下日语句子的语法。

要分析的句子：{用户选中的句子}

请从以下方面分析：
1. 整体句子结构和意思
2. 关键语法点逐一讲解
3. 类似表达方式
4. 使用场景和注意事项

请使用 Markdown 格式组织你的回答，使其清晰易读。
```

## 6. 词典与数据源

### 6.1 内置日语词典

- **推荐数据源**：[JMdict](https://www.edrdg.org/jmdict/j_jmdict.html)（开源日英词典）或 [JMdict-zh](https://github.com/jiang-wei/local-jmdict-zh)（日中翻译版本）
- **加载方式**：以 JSON 格式加载，实现本地快速查词（无需网络）
- **数据预处理**：提取常用词汇（约 2–5 万条），压缩后嵌入应用

### 6.2 词典展示内容

| 字段 | 说明 |
|------|------|
| 单词原形 | 辞書形（dictionary form） |
| 读音 | 平假名 / 罗马音 |
| 词性 | 名詞、動詞（自/他）、形容詞、副詞 等 |
| 中文释义 | 主要义项 |
| 常见用法 | 搭配短语或例句 |

### 6.3 备用方案

- 若本地词典未命中，可通过 AI 查询单词解释（复用 API）
- 需在 UI 中提示用户"词典未收录，是否使用 AI 查询？"
- AI 查询会消耗配额，需在配置页提示用户

## 8. 补充说明

### 8.1 数据存储

- 所有本地存储的会话数据以 JSON 序列化保存
- 注意转义特殊字符（如日文引号、换行符等）
- 监控 localStorage 容量（通常 5–10MB），接近上限时提示用户清理旧数据
- 超大对话历史可迁移至 IndexedDB

### 8.2 响应式设计

- 桌面端（≥1024px）为优先设计目标
- 移动端至少保证基础可用性（单栏布局，左右面板全屏覆盖）
- 平板端可折叠侧栏

### 8.3 扩展性

- **角色扩展**：初期实现 4–6 个角色，架构支持后续扩展
- **难度扩展**：初期实现 3 个难度等级（初级/中级/高级）
- **词典扩展**：支持后续替换或扩展词典数据源
- **多语言**：预留 i18n 接口，未来可添加其他语言学习支持

### 8.4 性能考虑

- 词典数据按需加载（懒加载），避免首屏加载过慢
- 聊天历史分页加载（虚拟滚动），避免大量 DOM 节点
- AI 请求添加防抖，避免重复请求
- 图片资源使用 SVG 图标，减少 HTTP 请求

### 8.5 安全提示

- 配置页面明确提示：API Key 仅保存在本地浏览器，不会上传至任何服务器
- 建议用户使用额度较小的 API Key 或设置使用限额
- 推荐使用 Vercel Edge Functions 代理方式以隐藏密钥
- 不在前端代码中硬编码任何密钥或敏感信息
